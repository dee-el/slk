# Architecture Decision Records

This directory contains tracked architecture decision records for `slk`.

## Workflow

1. Write or update an ADR before every non-trivial change.
2. Include context, decision, file-level implementation details, verification,
   rollout, risks, alternatives, and open questions in the ADR.
3. Mark the ADR `Accepted` before implementation starts.
4. Update the ADR when the implementation decision changes.
5. Commit ADRs with repository history so code and rationale stay together.

## Statuses

- `Proposed`: documented but not approved for implementation.
- `Accepted`: approved and current.
- `Superseded`: replaced by another ADR; retain for history.
- `Rejected`: considered but intentionally not adopted.
